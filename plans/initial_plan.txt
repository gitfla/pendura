Build a stateless MVP web app in Next.js + React + TypeScript + Tailwind for placing a painting photo onto a wall photo.

Requirements:

* Multi-step flow: upload wall photo, upload painting photo, crop painting photo, place painting on wall, optional perspective corner adjustment, render final PNG.
* No auth, no database, no permanent storage.
* Use client-side state for the editor.
* Crop step must be mandatory and use a simple freeform rectangular crop UI.
* Placement step must support drag, resize, rotate.
* Add an advanced “Adjust corners” mode with 4 draggable corner handles.
* Create a Next.js API route /api/render that accepts the wall image, cropped painting image, quad coordinates, and returns a composited PNG.
* Use a Python OpenCV renderer for perspective transform, shadow generation, and compositing.
* Make the UI mobile-aware, with larger handles and explicit edit modes on smaller screens.
* Keep the architecture simple and production-minded, but optimized for fast MVP implementation.


Final v1 decisions
Stack: Next.js monolith, same style as Comidinha
Rendering: Python + OpenCV
Storage/DB: none for now
Painting input: regular painting photos
Crop step: mandatory, simple freeform rectangular crop
Placement: drag, resize, rotate
Perspective adjustment: optional 4-corner mode
Platforms: mobile-aware from day one, but editor UX can be stronger on desktop
Output: PNG only
AI: not in v1

Recommended architecture
Frontend

Use your existing pattern:

Next.js App Router
React
TypeScript
Tailwind
local component state
localStorage only if you want very light recovery
Backend

Inside the Next app:

/api/render
optional /api/health

The backend should stay thin:

validate request
pass images + quad + options to Python renderer
return PNG bytes
Renderer

Separate Python script or tiny local service:

read wall image
read cropped painting image
compute perspective transform
warp painting into destination quad
generate soft shadow
composite result
return PNG

Acceptance criteria for v1

You should be able to say yes to all of these:

user can upload a wall photo
user can upload a painting photo
user must crop painting before continuing
user can place painting on wall
user can resize and rotate it
user can optionally adjust all 4 corners
app produces a believable final PNG
no account is required
no data is permanently stored

If all of that works, v1 is valid.

Tradeoff notes you should keep in mind
API routes in-app

Good now because:

fewer moving parts
faster iteration
easier local dev

Only split later if:

render times get annoying
AI enhancement becomes async
you need queueing/retries
deploy/runtime constraints become painful
SSR vs SPA

For this editor:

treat it mostly like a client-side tool
use Next because it matches your existing stack and makes routing/API easy
don’t over-optimize SSR for the editor itself

So effectively:

landing/app shell: standard Next
editor: SPA-like client interaction
